    AWS EC2 Sandbox Guide¶

    In this course, we will be using Amazon Web Services (AWS) to run GPU-based workloads. Specifically, you will be working with AWS EC2 instances that are provisioned through Innovation Sandboxes, which are pre-configured AWS environments provided by the university that allow you to launch and manage cloud resources (such as EC2 instances) in an isolated, temporary workspace. Each sandbox is tied to a lease, giving you controlled, time-limited access to compute resources for course assignments.

    Warning

    Authorized Use Only

    The AWS GPU instances provided through this course are strictly for completing course-related assignments and running instructor-provided code.

    Any use of these resources for non-course purposes (including but not limited to personal projects, cryptocurrency mining, external research, or unrelated compute tasks) is strictly prohibited.

    Misuse of these resources will be taken very seriously and may result in severe academic consequences, including:

    • An automatic failure in the course

    • Referral for additional disciplinary action under university academic policies

    These systems are monitored. You are responsible for ensuring your usage complies fully with course guidelines.

    What is an EC2 Instance?¶

    An EC2 (Elastic Compute Cloud) instance is a virtual machine in the cloud that allows you to run programs on remote hardware. In our case, these instances are equipped with GPUs and preconfigured for CUDA development.

    Each EC2 instance is provided to you through an Innovation Sandbox lease.

    Important

    • Each lease lasts 7 days.

    • You must request a new lease for each assignment.

    • The instructor will approve lease requests within 24 hours.

    • Once your lease expires, your instance is terminated.

    • You should immediately request a new lease after your current one ends.

    What is a Session?¶

    When working with an AWS EC2 instance, a session refers to an active connection between your local machine and the remote instance. In this course, sessions are established using the AWS Session Manager, which allows you to securely access your instance through the terminal without needing SSH keys or opening network ports.

    The Session Manager is part of AWS Systems Manager (SSM) and acts as a bridge between your local environment and the instance. When you run the command:

    aws ssm start-session --target <INSTANCE-ID> --profile isb-sandbox

    you are creating a temporary interactive shell on the remote machine. This session remains active until you disconnect or it times out due to inactivity.

    Note

    • A session is not the same as the instance itself the instance continues running even after your session ends.

    • You can start a new session at any time as long as your instance is still active.

    Prerequisites¶

    Before accessing your instance from your local machine, you must install the required AWS tools and ssh key for connecting to a session.

    AWS CLI v2¶

    Install the AWS CLI:

    • macOS:

      brew install awscli

    • Linux: x86_64

       curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install
rm -rf awscliv2.zip aws
aws --version

    • Linux: ARM64

       curl "https://awscli.amazonaws.com/awscli-exe-linux-aarch64.zip" -o "awscliv2.zip"
unzip awscliv2.zip
sudo ./aws/install
rm -rf awscliv2.zip aws
aws --version

    • Windows:

      Download the MSI installer from: https://aws.amazon.com/cli/

    Verify installation in a terminal:

    aws --version

    Session Manager Plugin¶

    Install the Session Manager Plugin:

    You can either install via package install from the AWS Website or via the terminal here:

    • Linux- Ubuntu/Debian (x86_64):

      curl "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/ubuntu_64bit/session-manager-plugin.deb" -o "session-manager-plugin.deb"
sudo dpkg -i session-manager-plugin.deb

    • Linux- Ubuntu/Debian (ARM64):

       curl "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/ubuntu_arm64/session-manager-plugin.deb" -o "session-manager-plugin.deb"
sudo dpkg -i session-manager-plugin.deb

    • macOS:

      curl "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/mac_arm64/session-manager-plugin.pkg" -o "$HOME/Downloads/session-manager-plugin.pkg"
sudo installer -pkg "$HOME/Downloads/session-manager-plugin.pkg" -target /
sudo ln -sf /usr/local/sessionmanagerplugin/bin/session-manager-plugin /usr/local/bin/session-manager-plugin

      Note

      If you are using an Intel Mac, replace mac_arm64 with mac in the URL.

    • Windows (PowerShell as Administrator):

      Invoke-WebRequest -Uri "https://s3.amazonaws.com/session-manager-downloads/plugin/latest/windows/SessionManagerPluginSetup.exe" -OutFile "$env:TEMP\SessionManagerPluginSetup.exe"
Start-Process "$env:TEMP\SessionManagerPluginSetup.exe" -ArgumentList '/quiet' -Wait

    Verify installation in terminal:

    session-manager-plugin --version

    SSH Access Setup (Linux/macOS)¶

    The following two sections explain how to set up SSH access to your AWS instance for both Linux/macOS. Windows users should skip to the next section titled: SSH Access Setup (Windows).

    Get the Class SSH Key (Linux/macOS)¶

    Your instructor will distribute a private key file through Ed discussion (see post “[IMPORTANT] AWS Sandbox Setup”). Save it to:

    ~/.ssh/aws_dummy_key

    This key does not grant access on its own. It only works in combination with your own AWS SSO login, so it is safe to use even though it is shared across the class.

    Restrict its permissions so SSH will accept it:

    chmod 600 ~/.ssh/aws_dummy_key

    Configure SSH to Tunnel Through SSM (Linux/macOS)¶

    Open (or create) the file ~/.ssh/config on your laptop and add the following block. The entire ProxyCommand must be on one line — do not let your editor wrap it across multiple lines.

    Host i-* mi-*
    ProxyCommand aws ssm start-session --target %h --document-name AWS-StartSSHSession --parameters portNumber=%p --profile isb-sandbox --region us-east-1
    User ubuntu
    IdentityFile ~/.ssh/aws_dummy_key
    StrictHostKeyChecking no

    Notes:

    • Replace us-east-1 with your sandbox’s actual region if different.

    • The Host i-* mi-* line means this rule applies any time you connect using an EC2 instance ID (which always starts with i-) as the hostname.

    This step is done once and will work for every future instance, since it matches on the pattern of an instance ID rather than one specific ID.

    SSH Access Setup (Windows)¶

    These steps assume you are working inside Git Bash, not PowerShell or cmd.exe. Open Git Bash from the Start menu before continuing.

    Get the Class SSH Key( Windows)¶

    Your instructor will distribute a private key file through the course LMS (not email or chat). In Git Bash, ~ maps to C:\Users\<YourName>, so save it to:

    ~/.ssh/aws_dummy_key

    This key does not grant access on its own. It only works in combination with your own AWS SSO login, so it is safe to use even though it is shared across the class.

    Windows OpenSSH will reject a private key that is readable by other users, so you must lock down its permissions. chmod does not work reliably for this on Windows — use icacls instead, run from PowerShell (not Git Bash) just for this one command:

    icacls C:\Users\<YourName>\.ssh\aws_dummy_key /inheritance:r
icacls C:\Users\<YourName>\.ssh\aws_dummy_key /grant:r "$($env:USERNAME):(R)"

    If you skip this step, SSH will refuse to use the key and report a “permissions too open” error.

    Configure SSH to Tunnel Through SSM (Windows)¶

    Open (or create) the file ~/.ssh/config in Git Bash (this is the same file as C:\Users\<YourName>\.ssh\config) and add the following block. The entire ProxyCommand must be on one line — do not let your editor wrap it across multiple lines.

    Host i-* mi-*
    ProxyCommand aws ssm start-session --target %h --document-name AWS-StartSSHSession --parameters portNumber=%p --profile isb-sandbox --region us-east-1
    User ubuntu
    IdentityFile ~/.ssh/aws_dummy_key
    StrictHostKeyChecking no

    This is the same syntax as macOS/Linux — Git Bash interprets the config file and runs the aws command the same way a Mac/Linux shell does.

    Notes:

    Note

    If you prefer native PowerShell instead of Git Bash, the ProxyCommand needs to be wrapped so Windows knows to run it as a PowerShell command, and paths should use Windows-style backslashes:

    Host i-* mi-*
    ProxyCommand powershell.exe -Command "aws ssm start-session --target %h --document-name AWS-StartSSHSession --parameters portNumber=%p --profile isb-sandbox --region us-east-1"
    User ubuntu
    IdentityFile C:\Users\YourName\.ssh\aws_dummy_key
    StrictHostKeyChecking no

    However, connect.sh in Part 2 is a bash script and will not run under plain PowerShell. If you go this route, you will need to either translate connect.sh into a PowerShell script yourself or run its commands manually. Using Git Bash avoids this entirely, which is why it’s the recommended path for this course.

    This step is done once and will work for every future instance, since it matches on the pattern of an instance ID rather than one specific ID.

    Requesting an AWS Sandbox Instance¶

    Your EC2 instance is not active for the entire course. Instead, you must request a sandbox lease for each assignment (every 7 days).

    To request a sandbox:

    1. Navigate to the sandbox portal:

      https://devgt5mpyksk0.cloudfront.net/

    2. Log in using your UChicago CNET credentials.

      Warning

      If you have trouble logging in, contact the instructor immediately.

    3. On the home page, click “Request a Lease”.

    4. Select the template:

      MPCS 52072 GPU Programming

    5. Accept the terms and conditions and click Next.

    6. On the Review & Submit page, click Submit.

      • The instructor will approve your request within 24 hours.

      • After approval, the instance will go through a provisioning stage.

      • Provisioning takes a few minutes and prepares your environment.

    7. Once provisioning is complete, your sandbox instance is ready to use.

    Configuring AWS SSO Authentication¶

    As of April 20, 2026, access to your instance is only available via the terminal. You must configure AWS SSO (Single Sign-On) before connecting.

    Note

    You only need to perform this setup once per sandbox lease.

    Step 1: Find Your Sandbox Account Number¶

    1. Return to the sandbox portal.

    2. Locate your active lease.

    3. Click “Login to account”.

    4. Note your sandbox account number (typically between 1–40).

    Step 2: Configure AWS SSO¶

    Open a terminal and run:

    aws configure sso

    Enter the following values:

    Prompt

    Value

    SSO session name

    uchicago

    SSO start URL

    https://uchicago.awsapps.com/start/

    SSO region

    us-east-1

    SSO registration scopes

    Press Enter (accept default)

    A browser window will open:

    • Log in with your UChicago credentials.

    • Approve the authorization request.

    Back in the terminal:

    Prompt

    Value

    Select account

    Your sandbox account

    Role

    Select available role (there should only be one to select)

    CLI default region

    us-east-1

    CLI default output format

    json

    CLI profile name

    isb-sandbox

    Important

    This profile is saved and will be reused for future commands within the same lease.

    Logging Into Your AWS Instance¶

    You will access your EC2 instance entirely through the terminal using AWS tools.

    Step 1: Log In¶

    Run the following command:

    aws sso login --profile isb-sandbox

    A browser window will open:

    • Log in using your UChicago credentials

    • Approve the authentication request

    Once completed, return to your terminal.

    Step 2: Find Your Instance ID¶

    Run the following command to locate your running instance:

    aws ec2 describe-instances \
  --filters "Name=instance-state-name,Values=running" \
  --query "Reservations[].Instances[].{ID:InstanceId,Name:Tags[?Key=='Name'].Value|[0]}" \
  --output table \
  --profile isb-sandbox

    Example output:

    ---------------------------------------------
|            DescribeInstances              |
+----------------------+--------------------+
|          ID          |       Name         |
+----------------------+--------------------+
|  i-0abc123def456789  |  sandbox-gpu-dev   |
+----------------------+--------------------+

    Copy the Instance ID (e.g., i-0abc123def456789).

    Important

    Save your Instance ID somewhere locally. You will need it every time you connect to your instance.

    Step 3: Connect via Session Manager (SSM)¶

    To connect your instance you need to start a session. Run the following command:

    aws ssm start-session --target <INSTANCE-ID> --profile isb-sandbox

    Replace <INSTANCE-ID> with your actual instance ID from Step 2.

    If successful, you are now connected to your EC2 instance. You will run the above command each time you want to login to your instance after authentication.

    Step 4: Switch to the Development User¶

    After connecting, switch to the development user:

    sudo su - ubuntu

    Note

    • You must run this command every time you log in.

    • The ubuntu user acts as your working environment (similar to your home directory on the CS Linux servers).

    GitHub Workflow (Critical)¶

    Before proceeding, it is extremely important that you understand how storage works on your AWS instance.

    Warning

    Your EC2 instance is temporary.

    • When your 7-day lease expires, the instance is permanently deleted.

    • All files stored on the instance will be lost.

    • There is no recovery mechanism.

    You must continuously push your work to GitHub.

    We will not be sympathetic to claims of lost work due to an expired instance. It is your responsibility to ensure your work is saved externally.

    Best practice: Push your work to GitHub every time you make meaningful progress.

    Using GitHub with HTTPS (Personal Access Token)¶

    Because your AWS instances are short-lived, using SSH keys is inconvenient since you would need to recreate them for every new lease. Instead, we will use HTTPS with a GitHub Personal Access Token (PAT).

    Think of the Personal Access Token as your password when interacting with GitHub.

    Important

    You should store your token securely on your local machine. You will use it whenever Git prompts you for a password.

    Step 1: Create a Personal Access Token¶

    1. Go to GitHub Settings:

      https://github.com/settings/tokens

    2. Click:

      “Generate new token” → “Generate new token (classic)”

    3. Configure the token:

      • Note: mpcs52072-gpu

      • Expiration: 90 days (or longer if preferred)

      • Scopes: - repo (full control of private repositories)

    4. Click Generate token.

    5. Copy the token immediately and store it somewhere safe.

      Warning

      You will not be able to see this token again after leaving the page.

    Cloning the Course Repository¶

    We will use HTTPS to clone repositories. Lets first clone the course repository

    Repository URL:

    https://github.com/mpcs52072-sum26/mpcs52072-sum26/tree/main#

    On your AWS instance, run:

    git clone https://github.com/mpcs52072-sum26/mpcs52072-sum26.git

    When prompted:

    • Username: your GitHub username

    • Password: paste your Personal Access Token (NOT your GitHub password)

    After cloning:

    cd mpcs52072-sum26

    You are now ready to access course materials.

    Note

    You will use your Personal Access Token for all Git operations: clone, pull, push, etc.

    Installing CUDA Libraries and Profiling Tools¶

    The AWS Deep Learning AMI includes some pre-installed tools, but it does not include the full CUDA development toolkit required for this course.

    You must install these tools once per sandbox lease using the provided script.

    Step 1: Navigate to the Setup Directory¶

    After cloning the course repository:

    cd mpcs52072-sum26/aws

    Step 2: Run the Installation Script¶

    bash toolkit_install.sh

    This script will install:

    • CUDA development libraries

    • Compiler tools (nvcc)

    • Profiling tools (e.g., Nsight Compute, Nsight Systems)

    Note

    The installation may take several minutes. Please be patient.

    Handling dpkg Lock Errors¶

    If you see an error such as:

    dpkg is locked by another process

    this means another system process is currently using the package manager.

    Important

    • Wait a few minutes

    • Re-run the script:

      bash toolkit_install.sh

    Step 3: Reboot the Instance¶

    After installation completes, reboot the instance:

    sudo reboot

    This will:

    • Temporarily disconnect your session

    • Restart the instance with the new tools configured

    Step 4: Reconnect to Your Instance in the Terminal¶

    After the instance restarts, you must reconnect:

    aws ssm start-session --target <INSTANCE-ID> --profile isb-sandbox

    Then switch back to the development user:

    sudo su - ubuntu

    Working with VS Code and AWS¶

    1. Open VS Code.

    2. Press Cmd+Shift+P (Mac) or Ctrl+Shift+P (Windows/Linux).

    3. Select Remote-SSH: Connect to Host.

    4. Type your instance ID (for example, i-0abc1234567890) and press Enter.

    VS Code will open a new window connected directly to your sandbox instance. You can now browse files, edit code, and use the integrated terminal — everything runs on the GPU instance itself.

    Daily Workflow¶

    AWS SSO authentication tokens expire after a few hours. You will need to re-authenticate and reconnect each day.

    Typical daily workflow:

    1. Log in:

      aws sso login --profile isb-sandbox

    2. Connect to your instance via the terminal or VS Code.

    3. Switch to the development user:

      sudo su - ubuntu

    4. Work on your assignments, and push your work to GitHub frequently.

      git add .
git commit -m "your message"
git push

    Tip

    If logging in through the terminal, To avoid retyping your Instance ID every time, you can store it as an environment variable:

    export INSTANCE_ID=i-0abc123def456789
aws ssm start-session --target $INSTANCE_ID --profile isb-sandbox

    General Tips¶

    • Working directory:

      Use:

      /home/ubuntu/

      for all of your projects.

    • Back up your work:

      Instance storage is temporary. Always push your work to GitHub regularly.

    • Editing files:

      • Use terminal editors like vim or nano

      • Or edit locally and sync using Git (recommended)

    Troubleshooting¶

    Problem

    Solution

    aws sso login fails

    Re-run aws configure sso from the setup section

    describe-instances returns empty

    Your instance may not be running. Contact the instructor

    start-session says “target is not connected”

    Wait 5–10 minutes after startup (SSM agent delay)

    session-manager-plugin: command not found

    Reinstall the Session Manager Plugin

    Session disconnects after idle time

    This is normal — reconnect using start-session
