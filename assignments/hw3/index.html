    Programming Problem: Convolution-Based Edge Detection¶

    Convolution is a common image processing operation used to detect edges, smooth images, or extract features. It works by sliding a filter (also called a kernel or stencil) over an image and computing a weighted sum of neighboring pixel values.

    Given an image I and a convolution kernel K of size n×n, the output at pixel position (x, y) is:

    \[\text{output}(x, y) = \sum_{i=-r}^{r} \sum_{j=-r}^{r} I(x+i, y+j) \cdot K(i+r, j+r)\]

    Where r = n // 2 is the stencil radius. At the edges, some of the neighborhood may fall outside the image bounds. In this assignment, you will handle this case using zero-padding— i.e., treat out-of-bounds pixels as having value 0.

    Tasks: Here is what you will need to accomplish for this assignment:

    1. Implement both global and shared memory convolution kernels. The global kernel will only use global memory and the shared memory convolution kernel will use a mixture of shared and global memory. You are required to use 2D grids and blocks for this assignment. Please note that the image data itself is still a 1D arrray, but you will need to use 2D indexing to access the image data in your kernels.

    2. Load the stencil from a file on the host

    3. Copy the stencil to device constant memory

    4. Use your existing image_to_grayscale_kern to convert the input image to grayscale. You are not required to update this kernel to use 2D indexing, but you can if you want to. You can also choose to write a new kernel that uses 2D indexing for the grayscale conversion if you prefer.

    5. Implement zero-padding in both kernels

    6. Dynamically allocate shared memory for the shared memory version

    7. Measure and report kernel execution time using CUDA events

    The following sections will provide guidance on how to implement these tasks.

    Stencil Input Format¶

    The stencil will be stored in a text file containing space-separated integers with one row per line. The number of rows must equal the number of columns (i.e., the stencil must be square). The first line will specify the dimension followed by the line representing the space-separated integers with one row per line.

    Example: A 3×3 Laplacian stencil (``data/stencil3x3.txt``)

     0 -1  0
-1  4 -1
 0 -1  0

    Your CUDA kernel must use __constant__ memory to store the stencil coefficients.

    Example:

    __constant__ int stencil[49];  // Supports up to 7×7

    Then copy data to the device like this:

    cudaMemcpyToSymbol(stencil, h_stencil, stencil_size * stencil_size * sizeof(int));

    You may assume the maximum stencil size is 7×7 (49 elements total). You may include a runtime check to validate that the loaded stencil is 3×3, 5×5, or 7×7 but this is not required. Inside the data directory, we have provide example stencils for all dimensions. We will only use these stencil files when grading.

    Shared Memory: Dynamic Allocation¶

    In your shared memory version of the convolution kernel, you will use dynamically allocated shared memory to load a tile and its halo.

    Declare shared memory like this:

    extern __shared__ uint8_t tile[];

    Then, compute the amount of shared memory needed during kernel launch:

    int shared_mem_size = (BLOCK_SIZE + 2 * radius) * (BLOCK_SIZE + 2 * radius);
edge_detect_shared<<<grid, block, shared_mem_size>>>(...);

    This ensures that the kernel has enough space to store the current tile of the image, including the halo around it needed for stencil operations.

    Preprocessing: Grayscale Conversion¶

    Before launching your convolution kernel, you must first convert the input image to grayscale on the GPU using your kernel from the previous homework assignment:

    image_to_grayscale_kern<<<grid, block>>>(...);

    Only the grayscale image should be passed into your convolution kernels. As mentioned above, you can choose to update your existing grayscale kernel to use 2D indexing; however, you are not required to do so.

    Measuring Kernel Execution Time with CUDA Events¶

    For this assignment, you will measure and report the execution time of both your global memory and shared memory convolution kernels using CUDA Events.

    CUDA events are a simple and effective way to measure elapsed time between two points in your GPU code.Although we will revisit CUDA Events in more detail in a later module (to explore advanced uses like stream synchronization and multi-kernel coordination), for now we will only use them for basic performance timing.

    Basic Timing Workflow¶

    Here’s how you can time a kernel using CUDA events:

    cudaEvent_t start, stop;
cudaEventCreate(&start);
cudaEventCreate(&stop);

cudaEventRecord(start);

// Kernel launch
convolution_kernel<<<grid, block>>>(...);

cudaEventRecord(stop);
cudaEventSynchronize(stop);

float elapsed_ms = 0.0f;
cudaEventElapsedTime(&elapsed_ms, start, stop);

printf("Kernel execution time: %.3f ms\n", elapsed_ms);

    You should wrap the timing around only the kernel launch, not memory allocations or file I/O.

    GPU Profiling with Nsight Compute (ncu)¶

    For this assignment, you are required to profile both your global memory and shared memory convolution kernels using Nsight Compute (ncu), and use the profiling data to support your answers in a report.md file. Timing numbers alone (e.g., from CUDA events) are not sufficient to answer the report questions. You need to explain why the two kernels perform differently, and that requires looking inside the kernel’s actual GPU execution behavior.

    You have not been given a breakdown of every ncu metric in this spec. Part of this assignment is learning to read and interpret the tool’s output yourself, the same way you would need to on the job. See the sections below for how to get started.

    You must use sudo and the full absolute path to the ncu binary on an AWS instance. For example, you can run as follow:

    sudo /usr/local/cuda-12.3/bin/ncu --set full --target-processes all ./edge_detect global data/stencil5x5.txt data/test_img.png output.png

    Learning to Read ncu Output¶

    A few starting points for learning what the metrics mean:

    • Nsight Compute prints OPT and INF notes directly under most sections of its terminal output. These are the tool’s own plain-English explanations of what’s going on, and they often tell you exactly where to look next in the report. Read these carefully before searching elsewhere.

    • The official Nsight Compute Profiling Guide has a full metrics reference: https://docs.nvidia.com/nsight-compute/ProfilingGuide/index.html Use Ctrl+F to look up any metric name you don’t recognize (e.g., “Achieved Occupancy”, “L1/TEX Hit Rate”, “Warp Cycles Per Issued Instruction”).

    • You can list every available metric name with a one-line description directly from the command line:

      ncu --query-metrics

      This is a long list which you don’t need to read all of it, but it’s useful for looking up something specific once you know its name.

    • Every ncu report is organized into named sections: GPU Speed Of Light, Compute Workload Analysis, Memory Workload Analysis, Scheduler Statistics, Warp State Statistics, Launch Statistics, Occupancy, Source Counters, and others. Get comfortable with what each section is broadly about before worrying about every individual metric inside it.

    Using ncu Without the GUI¶

    You are accessing your AWS instance remotely through VS Code, so there is no graphical display attached to the machine your kernels run on. This means ncu-ui (the Nsight Compute GUI) is not something you can open directly on the instance, even though the ncu command-line tool itself is installed and fully functional. For this assignment, do all of your profiling entirely from the command line — everything you need for the report can be obtained this way.

    Note

    If you’re curious, it is technically possible to view .ncu-rep files in the GUI by copying the file to your local machine (e.g., via scp or the VS Code file explorer’s download option) and opening it in a locally installed copy of Nsight Compute. This is not required for this assignment. The command-line output contains everything you need but may be a nice way to visually explore a report if you want to.

    Profile and save a report in one step:

    ncu --set full -o report_global_7x7 ./edge_detect global data/stencil7x7.txt data/test_img.png output.png

    This prints the full text report to your terminal and saves a report_global_7x7.ncu-rep file that you can re-open later or include in your submission.

    Re-print a saved report’s text output later (e.g., to paste into report.md, or to re-check something without re-running the program):

    ncu --import report_global_7x7.ncu-rep

    Focus on one section at a time, to cut down on output while you’re exploring:

    ncu --set full --section Occupancy ./edge_detect global data/stencil7x7.txt data/test_img.png output.png
ncu --set full --section WarpStateStats ./edge_detect global data/stencil7x7.txt data/test_img.png output.png

    Use ncu --list-sections to see all available section names.

    Redirect output to a file so you can search through it or paste it into your report:

    ncu --set full ./edge_detect global data/stencil7x7.txt data/test_img.png output.png > global_7x7.txt
grep -A 5 "Occupancy" global_7x7.txt

    Report (Submit as report.md)¶

    Answer the following:

    1. What is the average runtime for each stencil size (3×3, 5×5, 7×7) using both global and shared memory? Use CUDA event timings as before.

    2. How does performance scale with stencil size, for both kernel versions? Is the scaling roughly linear, worse than linear, or something else? Relate this to how much work per pixel increases as radius grows (hint: how many multiply-adds does a single output pixel require for an n×n stencil?).

    3. What speedup (if any) does shared memory provide, and does that speedup change as stencil size increases? Support your answer with specific ncu metrics from your saved reports — cite the metric name, its value in each version, and explain in your own words what that metric actually measures.

    4. If the global memory version outperforms the shared memory version in any of your test cases, explain why using specific ncu evidence. Do not just guess. Run ncu --set full on both versions and examine the full report output. Identify at least two sections of the report (e.g., relating to occupancy, memory caching, or warp stalls) that show a measurable difference between the two kernels, and explain what each metric you cite measures and why the difference matters for performance.

    5. Examine the Launch Statistics and Occupancy sections of your reports. What is Waves Per SM for your kernels on the test image? Is your 2D grid large enough to fill the GPU (i.e., is Waves Per SM comfortably above 1), or are you leaving SMs idle? If you tested a larger image (15MB+, as suggested below), how did Waves Per SM and the GPU Speed Of Light throughput numbers change compared to the smaller image?

    6. Look at Registers Per Thread and the four “Block Limit” values in the Occupancy section for both kernel versions. Is register usage the binding constraint on occupancy for either kernel, or is something else (shared memory, warps, SM count) the limiting factor? Explain your reasoning using the Block Limit values.

    7. What implementation issues did you encounter with:

      • Loading the stencil from file?

      • Allocating shared memory (particularly using dynamic shared memory with 2D tiles + halo)?

      • Managing kernel launch arguments?

      • Using constant memory?

      • Interpreting any Nsight Compute metric that was confusing or unexpected, and how you figured out what it meant?

    You are not required to produce graphs for this assignment, but you are required to cite specific metric names and values from your ncu reports to support your answers to questions 3–6. Please provide a brief overview of what’s happening in your implementation and execution when answering the above questions.

    Also, I recommend downloading even larger images (e.g., 15MB or greater) and considering how that might affect the timing and profiling results.
