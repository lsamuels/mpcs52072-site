    In this assignment, you will extend your convolution-based edge detection implementation by introducing thread coarsening, and then use profiling tools to analyze its performance.

    Specifically, you will:

    1. Implement thread coarsening on your shared memory kernel

    2. Profile your kernels using NVIDIA Nsight Compute (ncu)

    3. Evaluate performance across multiple stencil sizes

    4. Analyze performance tradeoffs using real profiling data

    You will perform experiments using the following stencil sizes:

    • 3x3

    • 5x5

    • 7x7

    Task 1 — Thread Coarsening¶

    In HW3 you already saw that increasing stencil size (radius) increases the number of multiply-adds per output pixel, and that shared memory’s advantage over global memory changes as that per-pixel work grows. Thread coarsening is a different lever: instead of changing how much data one thread reads, it changes how much work one thread does, which is how many output pixels a single thread is responsible for producing.

    Coarsening trades occupancy (fewer, “heavier” threads means fewer resident warps per SM) against amortized overhead (each thread’s halo-loading and index arithmetic is now spread across more useful work). Unlike the radius sweep, where the “cost” side of the tradeoff was largely about redundant memory traffic, the “cost” side of coarsening is primarily about running out of registers or occupancy headroom. You will use this assignment to make that distinction concrete with your own profiling data.

    Please watch the videos for week 4 where I explain thread coarsening.

    Implementing edge_detect_coarsened¶

    Add a third kernel, edge_detect_coarsened, alongside your existing edge_detect_global and edge_detect_shared kernels.

    Coarsening axis: each thread computes coarse_factor consecutive output pixels along the x-direction (i.e., along a row), while still covering a single row of the block in y. This keeps the tiling logic close to your existing shared-memory kernel rather than requiring a full 2D coarsening scheme.

    Shared memory sizing: your HW3 shared kernel allocated a tile of size (BLOCK_SIZE + 2*radius) x (BLOCK_SIZE + 2*radius). For the coarsened kernel, each block now covers BLOCK_SIZE_X * coarsening output columns, so the tile becomes:

    int shared_mem_size =
    (BLOCK_SIZE_X * coarse_factor + 2 * radius) *
    (BLOCK_SIZE_Y + 2 * radius);
edge_detect_coarsened<<<grid, block, shared_mem_size>>>(...);

    Declare shared memory the same way you did in HW3:

    extern __shared__ uint8_t tile[];

    Stencil access: continue reading stencil coefficients from __constant__ memory exactly as in HW3, which coarsening does not change how the stencil itself is stored or loaded.Your implementation must work for all stencil sizes (3x3, 5x5, 7x7)

    Zero-padding: your zero-padding boundary handling from HW3 still applies at the edges of the image; make sure it’s still correct at the edges of each thread’s set of coarsened output pixels, not just the first one.

    Correctness requirement: the coarsened kernel’s output image must be pixel-identical to your global and shared kernels’ output, for every stencil size and every coarsening factor you test.

    Program Usage: Command-Line Interface¶

    Extend your CLI to accept the coarsened mode and an optional coarsening factor:

    ./edge_detect global|shared|coarsened stencil.txt input.png output.png [coarsening]

    • coarsening is only meaningful for the coarsened mode. Default to 1 if omitted.

    • As in HW3, you do not need argument-validation code, which you can assume valid arguments matching this usage.

    Example output:

    $ ./edge_detect coarsened data/stencil5x5.txt data/test_img.png output.png 4
coarsened memory version (stencil 5x5, coarsening 4): 2.048 ms

$ ./edge_detect shared data/stencil5x5.txt data/test_img.png output.png
shared memory version (stencil 5x5): 3.215 ms

    Continue measuring kernel execution time with CUDA events exactly as in HW3, which you wrap the timing around the kernel launch only.

    Task 2 — GPU Profiling with Nsight Compute (ncu)¶

    As in HW3, you must profile using ncu from the command line (no GUI on the AWS instance), and your report answers must be grounded in specific metric names and values. The timing numbers alone are not sufficient.

    Required profiling runs¶

    At minimum, generate and save the following reports:

    • report_coarsened1_3x3.ncu-rep, report_coarsened2_3x3.ncu-rep, report_coarsened4_3x3.ncu-rep, report_coarsened8_3x3.ncu-rep

    • report_coarsened1_7x7.ncu-rep, report_coarsened2_7x7.ncu-rep, report_coarsened4_7x7.ncu-rep, report_coarsened8_7x7.ncu-rep

    (i.e., a coarsening sweep of 1, 2, 4, 8 at both the smallest and largest stencil size.) You are encouraged to also profile 5×5 and/or additional coarsening factors (16, 32) if your report would benefit from more data points.

    For each run, pay particular attention to metrics from the Occupancy and Launch Statistics sections: specifically Achieved Occupancy, Registers Per Thread, and the Block Limit values (registers, shared memory, warps, blocks) since these are what will let you explain why a given coarsening factor helps or hurts, not just that it does.

    Example:

    sudo /usr/local/cuda-12.3/bin/ncu --set full --target-processes all -o report_coarsened4_7x7 ./edge_detect coarsened data/stencil7x7.txt data/test_img.png output.png 4

    Report ( report.md)¶

    Answer the following questions and place it in a report.md file in your repository:

    1. What is the average runtime for the coarsened kernel at coarsening factors 1, 2, 4, 8 for both the 3×3 and 7×7 stencils? Use CUDA event timings.

    2. For each stencil size, at what coarsening factor does runtime stop improving or start getting worse? Is the falloff point the same for 3×3 and 7×7, or does it shift? State the specific factor for each.

    3. Explain the falloff you identified in Q2 using your ncu occupancy and register data and cite the metric name and value at the falloff point and the point just before it. Specifically: does Achieved Occupancy drop as coarsening increases? Does Registers Per Thread grow? Which Block Limit value becomes the binding constraint?

    4. Compare the coarsening tradeoff to the stencil-size tradeoff from HW3. Both produce a “there’s a sweet spot” result, but for different underlying reasons. In 2-3 sentences, explain what limits performance as stencil size grows (revisit your HW3 answers) versus what limits performance as coarsening grows, and why these are different mechanisms even though the resulting charts look similar.

    5. Does the best coarsening factor depend on stencil size? If your answer to Q2 showed a shift between 3×3 and 7×7, propose an explanation grounded in your metrics — larger stencils mean larger shared-memory tiles per block, which interacts with coarsening’s own shared-memory growth. Does shared-memory usage, rather than registers, become the binding constraint at larger stencil sizes? Support your answer with the relevant Block Limit value.

    6. What implementation issues did you encounter extending your shared kernel to support coarsening particularly around shared memory tile sizing, zero-padding at the edges of a thread’s coarsened output span, or interpreting any new ncu metric?

    As in HW3, you are not required to produce graphs, but you must cite specific metric names and values from your saved ncu reports to support Q3-Q5.
