    Problem 1: Estimating Pi using the Leibniz formula¶

    Inside the p1 directory, open the file called pi.cu and write a CUDA program that uses an infinite series method to estimate pi. One of the most widely known infinite series that can be used for this is the Leibniz formula, as described here:

    • How To Make Pi

    Your program should use this formula, and must use the following usage and have the these required command-line arguments:

    Usage: pi terms num_blocks
terms = the number of iterations (i.e., intervals) to perform
num_blocks = the number of blocks to launch

    The program should read in the terms and num_blocks argument argument and only print the estimate of pi. The program will run in two phases:

    Phase 1: Calculate local estimates¶

    Define a kernel named compute_estimates that launches with a grid dimension equal to num_blocks and a block dimension of one. The single thread in each block computes its own local estimate of pi and stores the result in an output array named estimates. The single thread in each block must perform an equal amount of terms computations. This means for each block you will need to compute its start (inclusive) and end (exclusive) indices based on how much work each block must compute (i.e., terms\num_blocks ). For example, if terms=1000 and num_blocks=4 then the chart shows the breakdown for each block’s start and end interval.

    Block ID

    Start (Inclusive)

    End (Exclusive)

    Number of iterations

    0

    0

    250

    250

    1

    250

    500

    250

    2

    500

    750

    250

    3

    750

    1000

    250

    If terms/num_blocks does not divide evenly then have one block perform the necessary additional iterations to equal terms.

    You have full control over the arguments passed to compute_estimates, how memory is allocated, and how the interval is divided among the blocks. Please keep in mind that we are not looking at performance in this first homework assignment, so you do not need to come up with the most optimal solution.

    Phase 2: Aggregation¶

    Each block will produce a partial local-sum of pi. After the compute_estimates completes, the host-side adds up the partial sums and outputs the final computation for pi.

    Requirements:

    1. All local-sum computations must be done on the GPU. The host-side code only reads in the data, prepares it to be transferred to the GPU, launches kernels, and finally aggregates the local-sums to output the final estimate of pi.

    2. Please make sure you are verifying your CUDA calls by checking the error codes as I described in the videos.

    3. Please make sure to print out pi to 10 decimal places so we can better check the precision.

    Assumptions: You can assume that terms and num_blocks will always be given as integers and be provided. Thus, you do not need to perform error checking on those arguments. You also will not need to print out the usage statement for this problem. The usage is given here for clarification purposes. Please note the given points will never exceed maxGridSize in x.

    Problem 2: 2D Translations¶

    2D translation is a process of moving an object from one position to another in a two dimensional plane. In this problem, you will write a program that performs 2D translation on a set of points using 3D homogeneous transformation matrices.

    What is 2D Translation with Homogeneous Coordinates?

    In computer graphics, homogeneous coordinates allow for the representation of translations, rotations, and other transformations using matrix multiplication. A 2D point \((x,y)\) can be represented in homogeneous coordinates as \((x,y,1)\).

    A 2D translation can be represented as a \(3x3\) matrix:

    \[\begin{split}\begin{bmatrix} 1 & 0 & t_x \\ 0 & 1 & t_y \\ 0 & 0 & 1 \end{bmatrix}\end{split}\]

    When this matrix multiplies the homogeneous coordinate \((x,y,1)\), the result is a translated point \((x',y',1)\):

    \[\begin{split}\begin{bmatrix} 1 & 0 & t_x \\ 0 & 1 & t_y \\ 0 & 0 & 1 \end{bmatrix} \begin{bmatrix} x \\ y \\ 1 \end{bmatrix} = \begin{bmatrix} x + t_x \\ y + t_y \\ 1 \end{bmatrix}\end{split}\]

    For example, consider we had the points \((1, 1)\), \((2,3)\), and \((3, 1)\). If we apply a translation vector \((2, 1)\) to this shape using the homogeneous transformation matrix, each point of the triangle will move 2 units to the right and 1 unit up. The new positions of the vertices will be \((3, 2)\), \((4, 4)\), and \((5, 2)\), respectively.

    Make sure you understand how to perform matrix and vector multiplication. You may want to do a google search as refresher on that formula.

    Task¶

    Inside the p2/translate.cu directory, implement a CUDA program that performs 2D translation on a set of points using 3D homogeneous transformation matrices. The program will take in three command line arguments: points_file, matrix_file, and iterations with the program having the following usage statement:

    Usage: translate points_file matrix_file iterations
   points_file - the filepath to the file containing all the points
   matrix_file - the filepath to the file containing the 3D matrix
   iterations  - an integer representing how many times to translate the points

    The points_file file has the following format

    • The first line represents the total number of points in the file.

    • All remaining lines represent a 2D point where each integer coordinate is separated by a single whitespace.

    Inside the p2 directory, there is a file named points-001.in that provides an example of this format:

    3
1 1
2 3
3 1

    The matrix_file contains a single \(3x3\) matrix, where each line represents a single row of integers, where the columns are separated by a single whitespace. For example, inside p2/matrix-001.in provides an example of this format:

    1 0 2
0 1 1
0 0 1

    Finally, iterations represents how many times to translate a point by the provided matrix. For example, if iterations=3 then given point \((1, 1)\) and a translation vector \((2, 1)\) then the following represents the final location for the given point:

    1. \((1, 1)\) + \((2, 1)\) = \((3, 2)\)

    2. \((3, 2)\) + \((2, 1)\) = \((5, 3)\)

    3. \((5, 3)\) + \((2, 1)\) = \((7, 4)\)

    The program outputs the final translated location for each point after applying the matrix to it iterations number of times. The order of the output must be the same order of how the points appeared in the original points_file. Each point in the output is on a single line and each coordinate is separated by a single whitespace.

    Requirements

    1. All computation must be done on the GPU. The host-side code only reads in the data, prepares it to be transferred to the GPU and finally prints out the final locations of the points.

    2. There can only be a single kernel launch in this program unlike in problem 1.

    3. How you format the data in memory and determine the grid and block dimensions is up to you.

    4. You must write the CUDA code for performing matrix and vector multiplication. You can not just retrieve the tx and ty values directly from the matrix and add it to the vector coordinates.

    5. Please make sure you are verifying your CUDA calls by checking the error codes as I described in the videos.

    To help think about what to make your block and grid sizes for problem 2. You may want to query the device for the maximum number of blocks and threads per block in CUDA.

    int device;
cudaDeviceProp prop;

// Get the current CUDA device
cudaGetDevice(&device);

// Get device properties
cudaGetDeviceProperties(&prop, device);

// Query max threads per block
int maxThreadsPerBlock = prop.maxThreadsPerBlock;

// Query max blocks per dimension (grid size)
int maxGridSizeX = prop.maxGridSize[0];
int maxGridSizeY = prop.maxGridSize[1];
int maxGridSizeZ = prop.maxGridSize[2];

    Assumptions: We will always provide translate with the correct number of arguments and valid input. The file arguments points_file and matrix_file will always be formatted as specified in the above description and iterations will always be a positive integer.

    Testing¶

    Inside each problem directory, there is a test-data directory that contains some sample test cases. You can run these test cases by running the provided run_tests.sh script. This script will run your program on each of the test cases and compare your output to the expected output. you’ll see the a directory test-data inside each problem folder. You must update each problem’s SBATCH px-submit.sh script to run the test cases on the cluster.

    Each test-*** is split into a .in and .ans file. The .in represents that command I used to run the program executable with any additional arguments necessary. The .ans shows the STDOUT after running the command (this is the answer).

    Before submitting, make sure you’ve added, committed, and pushed all your code to GitHub. You must submit your final work through Gradescope (linked from our Canvas site) in the “Homework #1” assignment page via two ways,
